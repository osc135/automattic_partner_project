# Architectural Decision Record

## 1. Two-Stage Generation: Design Tokens, Not Markup

**Decision:** The AI generates a JSON design specification (colors, fonts, copy, section content) — never WordPress block markup directly. A deterministic template engine (`theme_builder.py`) then assembles that spec into valid block HTML.

**Alternatives considered:**
- **AI generates raw block markup.** Faster to prototype but wildly unreliable. Block comments have strict syntax (`<!-- wp:columns {"layout":{"type":"flex"}} -->`), and even small hallucinations — a missing closing tag, an invented attribute — produce "Block contains unexpected content" errors in the editor. Every generation would be a coin flip.
- **Hybrid: AI generates markup, post-processor fixes it.** Explored briefly, abandoned. The fix surface is too large: you'd need a full block parser, and patching one broken nesting level often breaks another.

**Why this works:** The AI is strong at creative decisions (naming, copy, color harmony, layout mood) and weak at producing byte-exact structured markup. By constraining AI output to JSON — where validation is trivial — and keeping all markup in tested templates, we get reliable output on every run. The templates are boring on purpose: they are the known-correct foundation that the design tokens bring to life.

**Trade-off:** Less layout variety per generation. Every theme shares the same structural skeleton (hero → middle sections → CTA → footer). A future version could expand the template library or let the AI select from multiple layout variants per section.

## 2. Design Archetypes (A–E)

**Decision:** The system prompt defines five archetypes (Editorial, Bold SaaS, Minimal Portfolio, Warm & Approachable, Clean Professional). The AI picks one based on the brief, and the archetype determines voice, visual direction, and which extra sections appear.

**Why:** Without archetypes, the AI tends to produce middle-of-the-road output — safe colors, generic copy, the same section order. Archetypes act as creative constraints: they push the model toward a specific visual identity and copy voice, which paradoxically produces more varied and distinctive results. "Design a theme for a bakery" gets warm/earthy/conversational output because archetype D kicks in, while "design a theme for a law firm" triggers archetype E's credibility-forward voice.

**Alternatives considered:**
- **No archetypes, just rely on the description.** Produces bland, similar themes regardless of input.
- **User selects archetype manually.** Adds friction to the UI without much benefit — the AI's classification is good enough, and the user already expresses intent through their description.

## 3. AI Provider: Gemini Primary, Anthropic Fallback

**Decision:** Gemini 2.5 Flash as the primary generation model, with Anthropic Claude as a fallback. OpenAI gpt-4o-mini for the cheap color extraction side-call.

**Why Gemini primary:**
- Fast and cost-effective for structured JSON generation.
- Reliably follows the output schema when given a clear system prompt.
- Good at creative copy within constraints.

**Why Anthropic fallback:**
- Different failure modes — if Gemini is down or rate-limited, Claude is likely still available.
- The provider abstraction (`AIProvider` ABC) makes this a config change, not a code change.

**Why OpenAI for color extraction:**
- Color palette inference from a description ("ocean vibes, warm sunset") is a small, focused task. gpt-4o-mini is the cheapest option that handles it reliably.
- Keeps the main generation call focused on design + copy without overloading the prompt with color theory.

**Provider abstraction:** All three providers implement the same `AIProvider.generate(system_prompt, user_prompt) -> str` interface. Swapping the primary provider is a one-line change in the factory function. This was a deliberate investment for exactly the reason the spec calls out — AI providers are a moving target.

## 4. Constrained Font Set

**Decision:** The theme supports exactly five Google Fonts: Montserrat, Schibsted Grotesk, Karla, DM Sans, and Open Sans.

**Why:** WordPress block themes load fonts via `functions.php`. Every font needs a specific Google Fonts slug with weight variants. If the AI picks an arbitrary font, we'd need to either validate it against the full Google Fonts catalog or risk a broken theme. By constraining to five well-tested options that cover a range of personalities (geometric, humanist, grotesque), we guarantee every generated theme loads its fonts correctly.

**Trade-off:** Less typographic variety. Acceptable for an MVP. A future version could expand the font catalog with a validated lookup table.

## 5. Template-Based Theme Assembly

**Decision:** Theme files (templates, patterns, parts) are built from Python string templates in `theme_builder.py`, not generated by the AI.

**Why:** WordPress block markup is unforgiving. A `<!-- wp:group -->` block must have matching `<!-- /wp:group -->` closers, correct JSON attributes, and proper nesting. Having the AI generate this is asking for breakage. Instead, `theme_builder.py` contains templates that are known to produce valid block markup — tested against a real WordPress instance. The AI's design tokens (colors, fonts, copy) are interpolated into these templates.

**This also solves the Custom HTML block requirement:** Since no template contains `<!-- wp:html -->`, and the AI never writes markup, Custom HTML blocks are structurally impossible. The validator confirms this as a safety net, but the architecture prevents it by design.

## 6. Validation Pipeline

**Decision:** Two-stage validation with fail-fast behavior.

1. **Design spec validation** (`validate_design_spec`): Checks the AI's JSON output has all required keys, valid hex colors, minimum content counts. Fails → 400 with details.
2. **Theme file validation** (`validate_theme_files`): After template assembly, checks all required files exist, theme.json parses, style.css has required headers, no Custom HTML blocks. Fails → 400 with details.

**Why two stages:** Stage 1 catches AI hallucination (missing keys, bad colors) before we spend time building files. Stage 2 catches template bugs and provides a safety net against regressions. Both stages return structured error details to the frontend — never a silent failure.

**Alternative considered:** A single post-assembly validation pass. Rejected because diagnosing "theme.json has wrong colors" is much harder after template assembly than catching "AI didn't return a colors object."

## 7. Stack Choice: FastAPI + React

**Decision:** Python/FastAPI backend, React/TypeScript/Vite frontend.

**Why FastAPI:**
- Native async support (useful for AI API calls, though currently sync).
- Pydantic for request validation — catches malformed briefs at the boundary.
- Lightweight, minimal boilerplate.

**Why React + Vite:**
- Wizard-style multi-step form is a natural fit for React's state model.
- Vite for fast dev iteration.
- Tailwind for rapid UI styling without fighting a component library.

**Alternative considered:** A single-page app that calls the AI directly from the browser. Rejected because the AI orchestration (prompt construction, validation, theme assembly, ZIP packaging) belongs on the server, and API keys should never be in client-side code.

## 8. Security Considerations

**Input sanitization:** All user-provided strings are stripped of HTML tags (`sanitize.py`) before being interpolated into prompts or theme files. This prevents:
- XSS via theme content (a malicious description ending up in `style.css` or pattern files).
- Prompt injection (HTML tags that could confuse the AI's output parsing).

**Theme slug generation:** User descriptions are converted to URL-safe slugs (lowercase, alphanumeric + hyphens, max 40 chars) to prevent path traversal or invalid directory names in the ZIP.

**API keys:** Stored in `.env`, loaded via `python-dotenv`, never exposed to the frontend. The `.env` file is gitignored.

**CORS:** Restricted to `localhost:5173` and `localhost:5174` (dev ports). A production deployment would tighten this further.

**What's not yet addressed:** Rate limiting on the `/api/generate` endpoint (an expensive operation that calls external APIs). A production version needs per-IP or per-session rate limiting to prevent abuse.

## 9. Process: Commits on Main

Early development was committed directly to `main` rather than through pull requests. This was a pragmatic choice during rapid prototyping but doesn't reflect how I'd work on a team codebase. The remaining deliverables (this ADR, documentation improvements, future features) use feature branches and PRs to demonstrate the intended workflow.
